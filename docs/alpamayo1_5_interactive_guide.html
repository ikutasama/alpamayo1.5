3. 模型结构：VLM 负责理解，Expert 负责动作

Alpamayo 1.5 可以看成两部分：一个会看图和生成推理的 VLM，一个把推理上下文变成连续驾驶动作的 action expert。

VLM Backbone 读图像、历史轨迹 token、prompt，自回归生成 CoC 推理，并提供 KV cache。

src/alpamayo1_5/models/base_model.py

Action Expert 复用 VLM text config，但删除 embed_tokens，直接吃 action embedding。

src/alpamayo1_5/models/alpamayo1_5.py

Action Space 把未来轨迹转换为 acceleration/curvature，也把采样 action 积分回轨迹。

src/alpamayo1_5/action_space/unicycle_accel_curvature.py

为什么 VLM 不直接生成轨迹 token？

Alpamayo 1.5 的主推理路径把“语言推理”和“连续动作生成”拆开：VLM 生成 CoC 并产生上下文 cache， action expert 在连续 action space 里通过 flow matching 采样。这样轨迹更容易满足连续性、平滑性和车辆动力学约束。

5. Action Expert：从噪声 action 到轨迹

这里是 Alpamayo 1.5 区别于纯语言生成轨迹的重点。

action space 来自 UnicycleAccelCurvatureActionSpace.get_action_space_dims()，返回 (64, 2)。

维度含义
64未来 64 个 waypoint，每 0.1 秒一个点
2第 0 维 acceleration，第 1 维 curvature，均为归一化值

FlowMatching.sample() 从 torch.randn(B*,64,2) 开始，沿着 expert 预测的 vector field 积分到真实 action 分布。

x = torch.randn(batch_size, 64, 2)
for t in linspace(0, 1):
    v = step_fn(x, t)
    x = x + dt * v

step_fn() 把 noisy action 投影成 token embedding，再让 expert transformer 结合 VLM cache 预测 vector field。

变量形状
x[B*,64,2]
future_token_embeds[B*,64,D]
expert last_hidden_state[B*,64,D]
pred[B*,64,2]

action_to_traj() 反归一化 acceleration/curvature，然后用 unicycle kinematics 积分出 x,y,yaw。

输出是 pred_xyz [B*,64,3] 和 pred_rot [B*,64,3,3]，最后 reshape 为采样轨迹集合。

6. SFT 训练：stage1 和 stage2 不一样

SFT 入口是 finetune/sft/train_hf.py。stage1 训练 VLM 离散轨迹 token 生成； stage2 训练 Alpamayo 1.5 action expert 的 flow matching head。

训练目标

让 VLM 学会 next-token prediction，尤其是生成 future trajectory 离散 token。

# finetune/sft/models/sft_base_model.py
outputs = self.vlm(input_ids=input_ids, labels=labels, **tokenized_data)
future_traj_loss = CE(outputs.logits, labels, traj_mask)
others_loss = CE(outputs.logits, labels, labels != IGNORE_INDEX)
loss = future_traj_loss + others_loss
变量形状说明
input_ids[B,L]历史和未来轨迹占位符已被真实离散 token 替换
labels_mask[B,L]通常只让 traj_future 参与 loss
logits[B,L,V]语言模型每个位置预测下一个 token
lossscalarfuture trajectory token CE + other token CE

训练目标

让 continuous action expert 学会从 noisy action 预测 flow matching vector field。

# finetune/sft/models/sft_alpamayo1_5.py
action = action_space.traj_to_action(ego_history, ego_future)  # [B,64,2]
training_data = diffusion.construct_training_data(action)
action_embeds = action_in_proj(noisy_x, timesteps)             # [B,64,D]
expert_outputs = expert(inputs_embeds=action_embeds, past_key_values=vlm_cache)
pred = action_out_proj(expert_outputs.last_hidden_state)       # [B,64,2]
loss = MSE(pred, x - noise)
变量形状说明
ego_future_xyz[B,G,64,3]GT 未来轨迹
action[B,64,2]acceleration + curvature
noisy_x[B,64,2]flow matching 训练输入
action_embeds[B,64,D]expert 输入 embedding
pred[B,64,2]预测 vector field
stage2 当前实现隐含假设 G=1。如果多个 trajectory group 被展平成 B*G，VLM cache 的 batch 维可能和 action batch 不一致。

finetune/sft/evaluate_hf.py 用 ReasoningSampler 采样轨迹，再用 DistanceMetrics 计算 ADE/minADE/corner distance。

输出形状
pred_xyz[B,N_set,N_s,64,3]
pred_rot[B,N_set,N_s,64,3,3]
ego_future_xyz[:, -1][B,64,3]

7. RL 训练：Cosmos-RL + GRPO

RL 入口是 finetune/rl/models/reasoning_vla/alpamayo_cosmos_rl_post_training_entry.py。 当前 RL 路径训练的是 VLM 离散生成路径，不直接训练 Alpamayo 1.5 continuous action expert。

函数输入输出
AlpamayoCosmosDataset.__getitem__idx{"idx","split"}
RVLADataPacker.get_rollout_inputidx stubTokensPrompt
ReasoningVlaVllmRollout.rollout_generationlist of promptslist of completion strings

配置里 rollout.batch_size=2，rollout.n_generation=12，所以一次 rollout batch 理想情况下产生 24 条 completion。

字段形状说明
tokenized_data.input_ids[B_rl,L_rl]prompt + completion + end token
labels_mask[B_rl,L_rl]只标记 completion 区域
logprob_masks[B_rl,L_rl]GRPO 只在这些 token 上取 logprob
logits[B_rl,L_rl,V]policy 模型输出
 
ratio = exp(logp_current - logp_old)
loss = -min(ratio * advantage, clipped_ratio * advantage)
loss += kl_beta * KL(current || reference)

直觉：同一个 prompt 的多条 completion 会被 reward 比较。reward 高的 completion 有正 advantage，模型会提高它的 token 概率；reward 低的 completion 有负 advantage，模型会降低它的 token 概率。

8. Reward：轨迹准确 + 驾驶舒适

当前默认 reward 是 demo 版本，但已经能跑通完整 RL 闭环。

ADE 轨迹误差

finetune/rl/rewards/traj_reward.py::calculate_ade()

只看 XY 平面，逐 waypoint 计算预测轨迹和 GT 轨迹的欧式距离，再取平均。

Comfort 舒适性

finetune/rl/rewards/comfort_reward.py::compute_comfort()

检查纵向/横向加速度、jerk、yaw rate、yaw acceleration 是否落在舒适阈值内。

# finetune/rl/rewards/aggregated_reward.py
if l2_dist < 3.0:
    reward = -traj_l2_weight * (l2_dist / 3.0) + comfort_weight * comfort_score
else:
    reward = -1.0
completion 如何变成轨迹？

decode_rollout_trajectory() 会从 completion 字符串中切出 <|traj_future_start|> 到 <|traj_future_end|> 之间的离散轨迹 token， 再调用 traj_tokenizer.decode() 得到 predicted_fut_xyz [G,64,3]。

11. 后续科研改进切入点

  • Reward 加强：在 ADE + comfort 外加入道路边界、碰撞风险、车道保持、终点误差、TTC 等指标。
  • Reasoning reward：解析 <|cot_start|>...<|cot_end|>，奖励推理是否解释了关键交通参与者和驾驶意图。
  • GRPO 稳定性：系统调 kl_beta、reference reset、positive NLL、completion 过滤和 reward normalization。
  • Hard case mining：让训练更关注高 ADE、低舒适性或推理-轨迹不一致的场景。
  • Action expert RL：这是更深入的课题，需要把 continuous action expert 的可微/可采样策略和 logprob 设计接进 RL，而不是只训练离散 VLM 路径。
工程上建议先改 reward 和 logging，因为它们最容易形成可靠实验闭环；等 reward 曲线、ADE、completion 解析都稳定后，再动模型结构和 RL loss。
