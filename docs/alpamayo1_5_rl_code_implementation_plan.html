Alpamayo 1.5 RL 代码改进实施蓝图

把 Causal Flow-GRPO 从论文想法压到代码层：reward 分解、CoC 解析、token mask 路由、per-token advantage、flow expert RL，以及每一步如何保持旧训练配置可跑。

0. 总原则

第一轮代码改造采用“默认关闭、逐项打开”的策略。也就是说，合入代码后不改 TOML 开关时，现有 ADE + comfort + GRPO 行为应保持一致；服务器只需要打开新配置项，才进入 CoC reward、advantage routing 或 flow expert RL。
先做 reward 和日志 低风险可快速验证

先让每条 rollout 有结构化 reward breakdown，观察分布，再让 trainer 消费它。

再做 token 级信用分配 核心创新GRPO 兼容

CoC token、轨迹 token、格式 token 不再共用同一个 advantage。

最后接 flow expert RL 高收益高调参成本

先冻结 VLM，只训练 action expert 的 LoRA/adapter 或小学习率全参。

修改 aggregated_reward.py

保持旧权重 traj_l2_weight、comfort_weight 必填；新权重默认 0 或开关关闭。这样旧 TOML 不会改变训练行为。

def compute_reward(...):
    traj = compute_traj_reward(...)
    comfort = compute_comfort_reward(...)

    if cfg.enable_coc_reward:
        coc = compute_coc_reward(to_be_evaluated, reference)
    else:
        coc = 0.0

    if cfg.enable_consistency_reward:
        consistency = compute_reasoning_action_consistency(...)
    else:
        consistency = 0.0

    final_reward = (
        w.traj_l2_weight * traj.reward
        + w.comfort_weight * comfort.reward
        + w.coc_weight * coc
        + w.consistency_weight * consistency
        + w.risk_weight * risk
    )

    return final_reward, {
        "reward": final_reward,
        "traj_reward": traj.reward,
        "traj_L2": traj.ade,
        "comfort_reward": comfort.reward,
        "coc_reward": coc,
        "consistency_reward": consistency,
        "risk_reward": risk,
        "format_ok": format_ok,
    }

3. Phase 2：CoC/轨迹 Token Mask 路由

当前 data_packer.py::get_policy_input() 在拼接 completion 后只创建了 labels_mask，表示哪些 token 参与 logprob loss。我们需要在同一位置创建更细的 mask。

修改点

函数 修改内容
get_policy_input() 保存 prompt_len，在 new_ids_row 上查找 <|cot_end|>、<|traj_future_start|>、<|traj_future_end|>，构造三类 mask。
policy_collate_fn() 把 tokenized_data 中的 mask concat 到 batch 顶层：coc_logprob_masks、traj_logprob_masks、format_logprob_masks。
临时 debug 删除当前 print("[DEBUG] ...")，改成受配置控制的 logger debug，避免 8 卡训练刷屏。 
def build_completion_token_masks(input_ids, prompt_len, special_ids):
    completion = torch.zeros_like(input_ids, dtype=torch.bool)
    completion[prompt_len:] = True

    coc = torch.zeros_like(completion)
    traj = torch.zeros_like(completion)
    fmt = torch.zeros_like(completion)

    cot_end = first_pos(input_ids, special_ids["cot_end"], start=prompt_len)
    traj_start = first_pos(input_ids, special_ids["traj_future_start"], start=prompt_len)
    traj_end = first_pos(input_ids, special_ids["traj_future_end"], start=prompt_len)

    if cot_end is not None:
        coc[prompt_len:cot_end + 1] = True
    if traj_start is not None and traj_end is not None:
        traj[traj_start:traj_end + 1] = True
    fmt[input_ids == special_ids["cot_end"]] = True
    fmt[input_ids == special_ids["traj_future_start"]] = True
    fmt[input_ids == special_ids["traj_future_end"]] = True

    return {
        "logprob_masks": completion,
        "coc_logprob_masks": coc & completion,
        "traj_logprob_masks": traj & completion,
        "format_logprob_masks": fmt & completion,
    }
注意：mask 的 shape 必须和 input_ids 一样，保持 compute_logprobs() 能直接消费。不要在这里 shift mask，沿用现有 trainer 的约定，避免 off-by-one。

4. Phase 3：Per-token Advantage GRPO

Cosmos 传给 trainer 的 rollout.advantage 是标量。为了稳定，不建议第一版改 Cosmos dispatcher，而是在 policy 侧根据 completion 重新计算 reward components，再在 trainer 里做组内归一化。

数据字段

字段 来源 shape / 类型 用途
reward_components get_policy_input() 调用 reward helper 重新计算 dict[str, float] 用于构造 component advantage 和日志。
coc_logprob_masks token mask builder [B,L] bool CoC token 的 advantage 路由。
traj_logprob_masks token mask builder [B,L] bool 轨迹 token 的 advantage 路由。
format_logprob_masks token mask builder [B,L] bool 格式 token 可用较小权重，防止破坏 special token 结构。

trainer 修改

新增私有函数，不直接改 compute_loss()，因为它已经接受 [B,L] advantage。

def _build_token_advantages(
    self,
    scalar_advantages,        # [B]
    user_mini_batch,          # contains masks
    component_advantages,     # dict[str, Tensor[B]]
    computed_max_len,
):
    base = scalar_advantages.unsqueeze(1).expand(-1, computed_max_len)
    if not cfg.advantage_routing.enable:
        return user_mini_batch["logprob_masks"] * base

    routed = torch.zeros_like(base)
    routed += user_mini_batch["coc_logprob_masks"] * component_advantages["coc"].unsqueeze(1)
    routed += user_mini_batch["traj_logprob_masks"] * component_advantages["traj"].unsqueeze(1)
    routed += user_mini_batch["format_logprob_masks"] * component_advantages["format"].unsqueeze(1)

    fallback = user_mini_batch["logprob_masks"] & ~(coc_mask | traj_mask | format_mask)
    routed += fallback * base
    return routed

组内归一化

rollouts 通常来自同一 prompt 的 n_generation 多个 completion。第一版按 (split, idx) 分组，对每个 component 做 z-score；如果组大小小于 2，则回退到 Cosmos 标量 advantage。

component_adv = normalize_by_prompt_group(
    payloads=payloads_list,
    components=[sample["reward_components"] for sample in processed_samples],
    fallback=advantages_t,
)

5. Phase 4：Flow Expert RL

这一阶段不要直接塞进现有 ReasoningVLA GRPO trainer。建议新增一个 flow RL 训练路径，先复用 SFT stage2 的 action expert forward，再引入 FPO-style objective。

文件 新增/修改 说明
finetune/rl/models/flow_action/ 新增目录 放 flow action expert RL 的 trainer、data packer、entry，避免污染 reasoning_vla 路径。
src/alpamayo1_5/diffusion/flow_matching.py 可选新增 compute_vector_field_mse() 把已有 compute_loss_from_pred() 的 reduction 拆开，得到 per-sample loss。
finetune/sft/models/sft_alpamayo1_5.py 参考，不建议直接大改 复用 stage2 的 action expert 调用方式：action_in_proj -> expert -> action_out_proj。 
old_loss = flow_mse_per_sample(v_old, target).detach()
new_loss = flow_mse_per_sample(v_theta, target)
ratio_proxy = torch.exp(torch.clamp(old_loss - new_loss, -5.0, 5.0))

surrogate_1 = ratio_proxy * advantage
surrogate_2 = ratio_proxy.clamp(1 - eps, 1 + eps) * advantage
loss_fpo = -torch.minimum(surrogate_1, surrogate_2).mean()
loss_anchor = beta_ref * mse(v_theta, v_ref.detach())
loss = loss_fpo + loss_anchor + beta_sft * new_loss.mean()
Flow RL 是第二轮实现重点。第一轮如果先把 CoC/traj token routing 跑通，论文就已经有可展示的新算法；Flow expert RL 再作为增强版本，能降低整体调试风险。

8. 第一轮我建议实际实现的范围

先实现 Commit 1-3：reward breakdown、CoC/consistency 的规则版本、token mask、per-token advantage routing。它们足以支撑 Causal Flow-GRPO 的离散版本论文实验，而且不会要求本机有 A100 或完整数据挂载。

第一轮不建议马上实现完整 Flow expert RL，因为它涉及 action expert 的采样、old/ref expert 缓存、per-sample flow loss、以及和 closed-loop reward 的耦合。等服务器上确认 Commit 1-3 的 reward 和训练曲线稳定后，再加 Flow RL，会明显减少反复部署和 debug。
